# Three.js 3D 区域与路径编辑器对齐方案

## 1. 目标与验收口径

目标是在 UrhoX 的三维场景中完整复现 Three.js 当前区域/路径编辑器的**可达功能、交互结果、状态反馈和生成契约**，同时保留 UrhoX 已有的三维实体覆盖层与相机手感。

运行边界：本项目是本地 UrhoXEditor / UrhoXRuntime 项目，不属于 TapTap Maker 绑定项目。本方案的构建、运行、截图和交互验收全部走本地 UrhoX；除非用户另行明确要求，不执行 Maker 初始化、绑定、提交或远程预览。

验收不以“看起来相似”为准，而以以下结果为准：

1. 相同初始区域和路径数据，经相同编辑动作后，房间、连接、门槽、折点、宽度、楼层、角色语义、房间组和楼梯状态一致。
2. 编辑动作完成后重新生成，场景中的区域、走廊和楼梯仍与编辑状态一致，不回弹、不被自动生成覆盖。
3. 右键菜单、快捷键、选择反馈、错误提示和危险操作确认均可到达。
4. 高 DPI 下 NanoVG 菜单、工具栏命中与鼠标射线不偏移。
5. Three.js 中只有绘制代码、没有入口的死分支不作为必须复刻的交互；UrhoX 已有且不冲突的直接“绘制房间/连接房间”工具保留为增强能力。

## 3. 当前 UrhoX 实现与数据流

### 3.1 入口和调用链

```text
ControlPanel
  -> DungeonApp:ToggleEditor / SetEditorMode
  -> ForgeCameraController:BeginEditView / UpdateEditView
  -> SceneLayoutEditor:Update + Render
  -> SceneLayoutEditor:Commit
  -> DungeonApp:Generate(editorEnabled=true)
  -> DungeonGenerator.Generate(editorRooms, editorEdges)
  -> MultiFloor.Build
  -> DungeonRenderer.Build
  -> SceneLayoutEditor:SyncDungeon
```

### 3.2 已经具备的区域能力

- 三维顶视正交编辑进入/退出过渡。
- 当前层区域覆盖、点击选择、空白取消选择。
- 拖动区域，四角和四边缩放，整数网格吸附。
- 锁定/解锁；锁定区域禁止移动、缩放和删除。
- 拖拽绘制新区域。
- 删除区域并重排连接端点。
- 房间组赋予及删除房间组后的引用清理。
- 跨二维/三维编辑模式同步当前层和选择。

### 3.3 已经具备的路径能力

- 选择路径和显示门槽/折点控制点。
- 拖动门槽、折点和整段路径。
- 门槽释放到另一房间时重新挂接端点。
- 路径宽度 1–6。
- 添加折点、删除整条路径。
- 房间移动/缩放时按路径长度权重自适应折点。
- 删除路径后标记断开区域为密室；重新连接后清理密室状态。
- 生成器把 `editorEdges` 当作权威连接，不会偷偷补回已删除路径。

### 3.4 当前结构风险

- `SceneLayoutEditor.lua` 已有 1083 行，继续堆叠会超过 1500 行；新增逻辑必须模块化。
- 房间 id 当前等于 Lua 数组索引，删除后会重排。内部可以继续使用该契约，但所有连接重排必须集中处理并有测试。
- 生成与编辑在每次提交后立即重建；复杂动作需要事务快照，避免半成品楼梯或菜单动作进入生成器。
- raw NanoVG 与物理鼠标坐标混用；工具栏已做 DPR 转换，但右键菜单也必须复用同一逻辑坐标层。

### 4.3 跨层路径（楼梯）

Three.js 当前把楼梯作为路径编辑器的一部分。要声称“区域和路径完整”，必须包含：

- 只能连接相邻楼层。
- 优先匹配上下层有足够重叠的房间。
- 没有合适房间时，经确认自动创建上下层成对楼梯间。
- 楼梯预览、确认、取消；取消恢复动作前完整编辑状态。
- 楼梯整体移动、90° 旋转、拖拽方向控制点。
- 旋转时自动扩展普通房间或重塑专用楼梯间，连接路径同步自适应。
- “稳定自动”和“锁定位置”两种模式。
- 固定锚点、方向、宽度、长度、平台深度和候选序号传入生成器。
- 冲突状态和原因反馈；冲突时禁止确认。
- 删除关键楼梯前做连通性检查并确认。
- 当前层显示上楼/下楼、对侧楼层和冲突状态。

UrhoX 生成器已有自动楼梯空间合同，但还没有编辑端 `stairSpec` 和固定候选。必须同时补编辑器和 `MultiFloor`，不能只画一个假楼梯控件。

### 4.4 Three.js 基线中不照抄的死分支

`drawEditor` 仍包含 `editor.drag.mode === 'draw'` 的绘制和落地代码，但当前 `initEditor` 没有任何入口创建该 drag 状态；实际新增区域入口是空白右键“添加区域”。因此：

- UrhoX 现有“绘制房间”工具继续保留。
- Three.js 的真实右键新增行为必须补齐。
- 不把 Three.js 的不可达 draw 分支误写成“基线已有拖绘入口”。

## 5. 分辨率、坐标和输入契约

本项目未指定固定设计分辨率，raw NanoVG 使用模式 B：系统逻辑分辨率 + DPR 修正。

```lua
local physicalWidth = graphics:GetWidth()
local physicalHeight = graphics:GetHeight()
local dpr = graphics:GetDPR()
local logicalWidth = physicalWidth / dpr
local logicalHeight = physicalHeight / dpr

nvgBeginFrame(vg, logicalWidth, logicalHeight, dpr)
```

输入契约：

- NanoVG 工具栏、菜单命中：`logical = physical / dpr`。
- `Camera:GetScreenRay`：继续使用物理宽高归一化，不能把已除 DPR 的坐标再次用于射线。
- 右键使用 `MOUSEB_RIGHT`，中键使用 `MOUSEB_MIDDLE`，禁止数字按钮常量。
- 所有菜单矩形保存在逻辑坐标；窗口变化后按当前逻辑宽高重新约束位置。
- UI 组件命中优先于场景编辑；打开模态框或指针位于 UI 上时结束/暂停编辑拖拽。

## 8. 测试与验收清单

### 8.1 自动测试

- 路径简化：重复点、同向共线、折返不误删。
- 控制点吸附：X/Y 可独立吸附，超出阈值只做网格吸附。
- 端点换房：不能连回自身或另一端房间，不能产生重复连接。
- 区域删除：连接端点重排正确，房间组/角色不串房。
- 角色唯一：起点和终点各最多一个。
- 移层：不会产生跨两层以上连接。
- 密室：删除路径后主连通分量选择顺序正确，重连后恢复。
- 楼梯目标：只能相邻层。
- 楼梯匹配与成对楼梯间：重叠、占用、无空间三种情况。
- 楼梯移动/旋转：锚点、上下平台和房间适配一致。
- 关键楼梯删除：能识别断开主连通图。
- 固定 `stairSpec`：生成器优先使用指定锚点和方向，冲突时报告失败而不是静默换位置。

### 8.2 手工交互

- 左键区域：选择、移动、四边/四角缩放。
- 空白右键：添加 12×9 区域。
- 区域右键：删除、锁定、移层、连接、起点、终点、密室、房间组。
- 路径左键：选择和拖整段。
- 门槽/折点：拖动、吸附、改挂端点。
- 路径右键：命中位置加折点、重置、宽度、删除。
- 折点/门槽右键：删除折点、重置门槽。
- 删除键：房间删除；普通路径有折点时先恢复自动形状。
- 楼梯：添加、自动匹配、成对楼梯间、预览、旋转、移动、锁定、确认、取消、关键删除确认。
- Escape：先关闭菜单/取消楼梯流程，再退出编辑。
- Tab：切换二维/三维且数据不丢。
- 1x/2x/3x DPR：工具栏、菜单和场景射线均命中正确。

## 9. 完成定义

只有同时满足以下条件才能标记完成：

1. 差距矩阵中“处理”为新增/补齐的项目全部落地。
2. 新增测试通过，原生成/渲染/自定义设置测试不回归。
3. 本机三维编辑逐项验证通过。
4. 本地 UrhoXRuntime 测试、主入口烟测和编辑模式截图验收通过。
5. 最终报告列出仍存在的差异；若楼梯或任一菜单动作未落地，不能写“完全对齐”。
