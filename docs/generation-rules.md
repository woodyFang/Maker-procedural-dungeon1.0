# 生成规则分类与执行约束

## 1. 文档定位

本文是项目生成规则的权威分类文档。规则只分为两类：

1. **通用规则**：不依赖题材，约束布局如何生成、结果是否有效、场景是否可玩、资产是否合格。
2. **特定规则**：依赖具体题材、空间用途或视觉表达，决定生成什么内容以及内容长什么样。

不再使用“题材的通用规则”这一分类。代码中的 `GenericThemeRules`、`generationMode = "generic"` 和 `roomGroups` 是兼容标识，不代表产品规则分类。

## 2. 通用规则

### 2.1 输入、种子与确定性

- 相同种子和相同参数必须生成相同布局、连接、装饰及结构哈希。
- 布局、拓扑、语义、特征、路径雕刻、装饰和名称使用相互隔离的随机流。
- 单层参数变化应尽量只影响目标楼层；跨层楼梯和相邻落点是明确允许变化的共享契约。
- 生成失败时可以使用派生种子重试，但必须返回有效结果或明确错误，不得静默接受无效结构。

### 2.2 空间布局

- 场景由楼层、区域、通路、门和垂直连接组成。
- 自动区域不能重叠，区域之间保留最小间距，最终位置对齐生成网格。
- 区域尺寸、数量和密度必须归一化到实现支持的有效范围。
- 自动布局可执行散布、分离和紧凑化；编辑器手工布局是权威数据，不得在用户提交后再次自动推开。
- 小型、中型和大型区域是通用规模分类，不属于任何题材。

当前实现的默认参数包括：区域尺寸 `5~24` 格、每层区域数量 `3~50`、区域边界间距 `6` 格，以及最多约 `300` 次分离迭代。这些是当前算法参数，不是题材定义。

### 2.3 拓扑与连通性

- 每层必要区域必须互相可达。
- 基础连接图必须保证连通，额外回环用于增加路线选择。
- 不允许重复边、自连接或非法端点。
- 大型布局应保留一定数量的叶节点，避免所有区域都退化成无层次的高连接图。
- 当前实现使用 Delaunay 候选边、最小生成树保证连通，并按回环率加入距离合适的额外连接；区域数不少于 20 时尽量保留至少 3 个叶节点。

### 2.4 楼层数量与多层结构

- 场景至少包含 1 层。
- **楼层数量不设固定最大值。**
- **建议不超过 6 层**，这是性能、生成耗时和编辑管理体验建议，不是合法性限制。
- 超过建议值时系统可以提示成本和风险，但不得拒绝、截断或偷偷改回 6 层。
- 每两个相邻楼层必须有可达的垂直连接；连接不得无理由跨层。
- 修改某层时应保留未修改楼层的区域图和装饰，只有楼梯落点及邻近设施允许因跨层契约调整。

### 2.5 高度、单位与垂直连接

- 所有长度、坐标和模型尺寸统一使用米。
- 程序化美术基准层高为 `5.0m`，当前允许配置的层高为 `2.5m~8.0m`。
- 结构纵向尺寸按 `floorHeight / 5.0` 缩放；家具与普通道具保持人体尺度。
- 楼梯总升高必须等于楼层高度，踏步高度乘踏步数量必须等于总升高。
- 楼梯平台、梯井、楼板开口和净空区域必须预留，普通路径和模型不得侵占。
- 当前支持直梯和 L 型楼梯；样式、转折数据及分段步数必须一致。

### 2.6 路径与可通行性

- 入口、关键区域和目标区域必须可达。
- 路径搜索必须考虑已有通路、转弯、反向移动、穿越其他区域、楼梯预留区和禁用格的代价。
- 路径应减少无意义折线和连续蛇形。
- 任何生成内容都不得堵住入口、门、主通道、楼梯、平台或必要开口。
- 普通连接的门点必须解析到所属区域的真实墙边界；门宽覆盖的切向单元必须全部合法，墙外 approach 区域必须连续开放。
- 自动门可以在保持目标方向优先的前提下回退到其他合法墙面；用户指定的门面不合法时必须报告冲突，不得静默换墙。
- 走廊雕刻必须包含门点与门外 approach 段，门框必须使用最终解析出的墙体接口位置。

### 2.7 空间语义、流程与难度

通用 PCG 可以提供不绑定题材的抽象流程角色，例如：

- 入口区域
- 普通区域
- 高难度区域
- 奖励区域
- 特殊功能区域
- 首领区域
- 隐藏区域

当前兼容角色为 `entrance`、`combat`、`elite`、`treasure`、`shrine`、`boss` 和 `secret`。入口通常位于流程起点，奖励倾向叶节点，高难度区域位于中后段，目标区域位于流程末段。难度可随楼层和图深度递增。

这些值只是图生成和流程安排使用的职责标签。例如 `boss` 表示流程末端或高难度目标职责，**不代表所有题材都拥有一个名为“Boss 房”的房间**；`teleport` 也不是通用房间。Boss 房、传送房及其他具体命名空间只有在当前题材特定规则明确生成，或用户在当前题材作用域内手动创建时才存在。

### 2.8 摆放与出生点

- 装饰和出生点必须位于合法可通行表面。
- 摆放必须避开门、通路、楼梯、特殊地形、已占用单元和区域边界净空。
- 模型之间不能非法重叠；空间不足时必须返回冲突，不得静默删减精确要求。
- 数量可按区域面积、难度、密度和特定房间规则计算。

### 2.9 模型、资源与几何验收

- 最终结果必须使用真实可导入模型，不接受占位模型或虚假资源路径。
- 模型必须具有稳定 `assetId`、边界盒、枢轴、前方向、碰撞、材质和性能元数据。
- 不得用模型名称猜测尺寸；必须使用资产元数据或真实包围盒。
- 地面、墙体和结构封层必须覆盖接缝，墙脚与地面必须保持有效水平和垂直重叠。
- 墙挂物不得超过最低墙顶，结构件不得越出合法楼层空间。

### 2.10 色板与最终画面验收

- 颜色使用 `#RRGGBB`，并按地面、通路、墙体、结构、强调物、织物和发光等语义角色组织。
- 派生色必须来自语义色板，不能绕过色板为同类资产任意写死孤立颜色。
- 亮度检查基于排除 UI 后的最终渲染画面，而不是只检查输入色值。
- 当前验收范围：平均亮度 `0.12~0.82`、压黑像素比例不超过 `8%`、过曝像素比例不超过 `3%`、自发光覆盖率不超过 `6%`。

### 2.11 通用结构色调执行

所有题材的地面、走廊、墙体和墙帽必须执行同一套结构色调流程：

- 结构材质保持中性灰，主题色通过实例顶点色表达，禁止有色材质与有色顶点色重复叠加。
- 基础色从当前主题的语义色板读取：`floor`、`corridor`、`wall`、`cap`。
- 地面统一叠加房间角色色、门口提亮、周边墙体暗化、棋盘变化和确定性随机变化。
- 墙体和墙帽统一叠加受控的随机变化；题材只能提供参数，不能在渲染器内绕过解析器硬编码结构颜色。
- `settingKey` 负责结构体系和基础光照预设，`themeKey` 负责语义色板、派生色及现有主题环境颜色；题材特定结构调制参数放在环境 profile 中。
- palette 可见度调制（相对灰度的饱和度、对比度和主题强调色混合）也属于通用结构色调流程；它只能由环境 profile 提供强度，不能在渲染器内写死医院或学校颜色。
- `Themes` / `PaletteData` 提供颜色，`EnvironmentProfiles` 提供调制强度，`ThemeToneRules` 执行颜色解析，`ExactGeometryBatcher` 写入顶点色，`DungeonApp` 只负责 fog、sky、ambient、sun 等环境光照。
- 配色分为两类：**默认配色**可跨所有题材复用；**题材配色**只适用于所属题材。二者都只提供色板与派生色，不改变题材的结构、道具、房间语义或丰富度规则。
- 面板必须将默认配色与当前题材配色分区展示；自定义配色也必须显式保存其分类，默认自定义配色可以跨题材使用。
- 选择默认配色时，`settingKey` 仍决定结构体系与题材资产；选择题材配色时，`themeKey` 同时提供该题材的专属色板与环境气氛。
- 结构色调规则必须同时适用于遗迹、医院和学校，并通过最终渲染画面的亮度、黑白比例、语义对比和自发光覆盖率验收。
- 题材可以使用不同的色板、模型、房间语义和光照气氛，但不得降低通用视觉质量基线；每个题材都必须提供关键房间强调、墙面装饰、地面/走廊填充和环境深度层的语义资产或明确关闭原因。
- 遗迹的丰富度不是遗迹专属规则。医院和学校应通过自己的医疗/校园资产执行相同的丰富度 pass，不能因为题材不同而退化成只有结构面和少量房间道具的空场景。

## 3. 特定规则

特定规则依赖题材、空间用途或视觉方案，包括：

- 学校的教室、图书馆、实验室、食堂和大厅等空间语义。
- 医院的病房、手术室、护士站、MRI 室和隔离区等空间语义。
- 某题材专属的模型、道具、材质、色板、灯光、墙体和门框样式。
- 冰湖、岩浆、墓地、根系等题材特征。
- 某类房间通常需要的素材池、密度、权重、朝向和宽松布局倾向。

特定规则只提供题材内容和参数；丰富度机制本身属于通用规则，当前包括：

- 房间内部布局模式：`grid`、`perimeter`、`ring`、`fill`、`focal` 和默认 `scatter`。
- 环境填充层：地面散布、墙面装饰/灯具、走廊标记、深度杂物和高难度区域强调物。
- 区域环境特征：房间特征选择、地形雕刻、表面覆盖、边缘附着物和局部斑块。

这些通用机制不包含遗迹、医院或学校的模型类型；模型、材质、启用条件、密度和数量由 `EnvironmentProfiles`、`ThemePacks` 或题材生成包作为数据提供。题材没有对应资产时可以关闭某层，不得复用不合语义的其他题材模型。特定规则可以改变内容选择和视觉表达，但不得违反通用连通性、可通行性、尺度、碰撞和验收规则。

## 4. 题材空间语义进入房间 UI

题材生成时识别出的专属空间语义必须作为**房间记录**加入当前题材的房间数据，并立即显示在现有房间 UI 中。例如学校题材生成后，可直接出现大厅、教室、图书馆、实验室和食堂。

这些房间记录需要：

- 绑定当前题材的稳定 `topicId`。
- 具有稳定 ID、名称、颜色、提示词和来源。
- 可携带流程角色、面积范围、最小/最大数量及专属素材规则。
- 可在房间 UI 中查看和编辑。
- 可通过现有区域分配下拉框赋予二维或三维场景中的具体区域。
- 重新生成题材时替换未锁定的 AI 房间，保留用户锁定或手工修改的房间。
- 保存后重新加载仍归属于正确题材。

当前持久化字段 `roomGroups` 和内部模块命名暂时保留以兼容已有数据；对用户统一显示为“房间”。不得为题材生成房间另建第二套数据源。

每个房间记录必须具有且只能具有一种作用域：

- 所有题材统一保存于 `customSettings`，遗迹、医院、学校只是随项目提供的初始题材记录，不是另一类“内置题材”。
- `baseSettingKey` 仅表示题材内部使用的结构、材质、灯光和生成执行体系，不形成独立的题材 UI 或数据生命周期。
- 所有房间都通过 `topicId` 绑定到所属题材；旧数据中的 `settingKey` 房间在加载时迁移到对应初始题材。
- 初始题材声明的默认特定房间必须在项目初始化、旧数据加载和云端数据覆盖后幂等地物化到对应题材的 `topicId` 下，并与用户创建的题材共用同一个房间 UI、区域分配和生成执行管线。
- 初始房间使用 `source = "seed"`，规则升级时可刷新未锁定的初始记录；用户手动创建或修改后使用 `source = "manual"` 并保留，不得被默认规则覆盖。
- 不允许没有 `topicId` 的全局房间。
- 手动新增的房间自动绑定创建时的当前题材；切换题材后不显示，也不能分配给其他题材的区域。
- 旧版本无作用域房间迁移到默认遗迹题材，以保留数据但停止跨题材泄漏。

## 5. 规则优先级

规则应用顺序固定为：

```text
区域细节覆盖
> 特定房间规则
> 特定题材规则
> 通用规则默认值
```

这里的高优先级只允许细化或覆盖内容选择。通用有效性约束始终生效，例如连通、门口避让、楼梯净空、模型碰撞和尺寸验收不能被特定规则关闭。

## 6. 术语约定

| 对外术语 | 含义 | 当前兼容实现 |
|---|---|---|
| 通用规则 | 跨题材的 PCG、有效性和验收规则 | `DungeonGenerator`、`MultiFloor`、`GeometryRules` 等 |
| 特定规则 | 某题材或某类房间的内容与视觉规则 | 题材包、题材生成计划和房间素材规则 |
| 房间 | 可复用且可赋给具体区域的题材空间语义 | 持久化字段暂为 `roomGroups` |
| 区域 | 场景中的具体生成实例 | `rooms` / editor room |

后续文档、UI 和需求描述均遵循以上术语。
